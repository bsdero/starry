## 1. question tool — TUI input collection

When the LLM calls the `question` tool the TUI must
pause and collect real user input before resuming.

### Current state
`question.execute()` returns a `user_input_required`
stub dict. The TUI renders a warning frame but does
not collect input; the tool loop continues immediately
with the empty stub as the tool result.

### Required work
- In `handle_ai_response()`, detect `tool_result`
  events where `name == "question"` and the result
  contains `type: "user_input_required"`.
- Pause the event loop using `asyncio.Queue`: the
  coroutine `await`s a queue that the TUI input
  handler writes to when the user submits a reply.
- Show a dedicated inline prompt frame (distinct from
  the normal chat input) labelled with each question.
- On user submission, inject the answer(s) back as the
  tool result string and resume the `chat_with_tools()`
  loop.
- Multiple questions in one call: collect all answers
  before resuming.

### Notes
- Architecture is analogous to item #4 (tool confirm).
  The `_tui_input_mode` flag should gain a third value
  `"question"`. The Session needs a `_question_queue`
  (separate from `_confirm_queue`).
- `question.execute()` should be made async and await
  the queue, similar to how task.execute() works.
  Alternatively, the TUI can detect `user_input_required`
  in the `tool_result` event and retroactively patch the
  result back into the conversation — but the queue
  approach keeps session.py self-contained.

## 9. User frame color persistence across mode switches

### Current state
User message frames are rendered with style classes
`line.uframe` and `line.ucontent`, which both map to
`mc` (the mode color) inside `build_style()`. When the
user switches mode, `app.style = build_style(new_mode)`
is called and ALL previously rendered frames repaint
using the new mode color — even messages typed in a
different mode.

### Root cause
The color is not baked into the buffer text at render
time. It lives in the style sheet, which is global and
mutable. prompt_toolkit re-applies the active style to
every classified token on every repaint.

### Required work
- Add two static style classes that do NOT change on
  mode switch:
  - `"line.uframe_plan"` → always `{MODE_CLR_PLAN}`
  - `"line.uframe_exec"` → always `{MODE_CLR_EXEC}`
  - Same pair for `"line.ucontent_plan"` /
    `"line.ucontent_exec"`.
  These must be defined in `build_style()` using the
  fixed palette values (`MODE_CLR_PLAN`,
  `MODE_CLR_EXEC`), not the variable `mc`.
- Introduce two new marker constants (alongside the
  existing `M_USER`, `M_AFRAME`, etc.) that encode
  which mode was active when the frame was built:
  - `M_UPLAN` — user frame written in plan mode
  - `M_UEXEC` — user frame written in execution mode
- Update `build_user_frame()` to accept the current
  mode and select the appropriate marker and class
  pair.
- Update every call-site of `build_user_frame()` to
  pass the current `_exec_mode`.
- Update the lexer (`DavyLexer.lex_document()`) to
  map `M_UPLAN` → `line.uframe_plan` /
  `line.ucontent_plan` and `M_UEXEC` →
  `line.uframe_exec` / `line.ucontent_exec`.
- Update `_replay_display_entry` for `"user"` entries:
  the display_log entry should store the mode that was
  active when the message was sent (add `"mode"` key
  to the user entry in `chat_with_tools` /
  `session.chat()`), and replay uses the correct
  marker.

## 10. Escape key — interrupt LLM thinking

### Current state
No key binding exists for Escape during an active LLM
call. The user cannot cancel a pending request.

### Required work
- Track the asyncio.Task returned by
  `asyncio.ensure_future(handle_ai_response(...))` in
  a module-level variable (e.g. `_ai_task`).
- Add a key binding for `Keys.Escape` (or `"escape"`)
  in the KeyBindings block.
  Condition: `_ai_task is not None and
  not _ai_task.done()`.
- On Escape:
  1. Call `_ai_task.cancel()`.
  2. Reset `telemetry.ai_status = "idle"`.
  3. Clear `_tui_input_mode = "chat"` (in case a
     confirm/question prompt was active).
  4. Clear `_da_session._confirm_queue` and
     `_da_session._question_queue` if set.
  5. Remove the spinner block from the scroll buffer
     (`replace_last_block(spin_lines, "")`).
  6. Show inline notification:
     `build_inline_notif("Interrupted", "✕")`.
  7. Set `_ai_task = None`.
- The `finally` block in `handle_ai_response()` must
  tolerate `asyncio.CancelledError` without raising
  (wrap cleanup in `try/except CancelledError`).

## 11. PgUp/PgDn — first keypress does not scroll

### Current state
The scrollable body window does not respond to the
first PgUp or PgDn keypress. Subsequent presses work
correctly.

### Investigation needed
The most likely causes are:
a) Focus is on the input `TextArea` by default.
   The first PgUp/PgDn is handled by the focused
   buffer (which has no content to scroll) and
   consumed before it reaches the scroll window.
b) The key binding for scroll is registered but the
   `main_buffer` Window is not focused, so
   prompt_toolkit routes the key to the focused widget
   first.
c) The scroll position is initialised to 0 and the
   first event triggers a focus transfer rather than
   an immediate scroll.

### Required work
- Audit the key bindings for PgUp/PgDn: confirm they
  target the `main_buffer` window directly (via
  `window.render_info` or explicit `focus` call).
- If the issue is (a)/(b): ensure the key bindings use
  `filter=Always()` or an explicit condition that does
  not require focus on the scroll window, and that they
  call `get_app().layout.focus(main_window)` before
  scrolling if needed — OR handle the scroll via
  `main_buffer` cursor movement directly without
  requiring focus transfer.
- If the issue is (c): verify the initial cursor
  position and scroll offset are set correctly after
  `main_buffer.set_document(...)`.
- Add a test case: open TUI, do not touch mouse or
  Tab, press PgDn immediately — it should scroll.
