## 1. question tool — TUI input collection

When the LLM calls the `question` tool the TUI must
pause and collect real user input before resuming.

### Current state
`question.execute()` returns a `user_input_required`
stub dict. The TUI renders a warning frame but does
not collect input; the tool loop continues immediately
with the empty stub as the tool result.

### Required work
- In `handle_ai_response()`, detect `tool_result`
  events where `name == "question"` and the result
  contains `type: "user_input_required"`.
- Pause the event loop using `asyncio.Queue`: the
  coroutine `await`s a queue that the TUI input
  handler writes to when the user submits a reply.
- Show a dedicated inline prompt frame (distinct from
  the normal chat input) labelled with each question.
- On user submission, inject the answer(s) back as the
  tool result string and resume the `chat_with_tools()`
  loop.
- Multiple questions in one call: collect all answers
  before resuming.

### Notes
- Architecture is analogous to item #4 (tool confirm).
  The `_tui_input_mode` flag should gain a third value
  `"question"`. The Session needs a `_question_queue`
  (separate from `_confirm_queue`).
- `question.execute()` should be made async and await
  the queue, similar to how task.execute() works.
  Alternatively, the TUI can detect `user_input_required`
  in the `tool_result` event and retroactively patch the
  result back into the conversation — but the queue
  approach keeps session.py self-contained.

## 11. PgUp/PgDn — first keypress does not scroll

### Current state
The scrollable body window does not respond to the
first PgUp or PgDn keypress. Subsequent presses work
correctly.

### Investigation needed
The most likely causes are:
a) Focus is on the input `TextArea` by default.
   The first PgUp/PgDn is handled by the focused
   buffer (which has no content to scroll) and
   consumed before it reaches the scroll window.
b) The key binding for scroll is registered but the
   `main_buffer` Window is not focused, so
   prompt_toolkit routes the key to the focused widget
   first.
c) The scroll position is initialised to 0 and the
   first event triggers a focus transfer rather than
   an immediate scroll.

### Required work
- Audit the key bindings for PgUp/PgDn: confirm they
  target the `main_buffer` window directly (via
  `window.render_info` or explicit `focus` call).
- If the issue is (a)/(b): ensure the key bindings use
  `filter=Always()` or an explicit condition that does
  not require focus on the scroll window, and that they
  call `get_app().layout.focus(main_window)` before
  scrolling if needed — OR handle the scroll via
  `main_buffer` cursor movement directly without
  requiring focus transfer.
- If the issue is (c): verify the initial cursor
  position and scroll offset are set correctly after
  `main_buffer.set_document(...)`.
- Add a test case: open TUI, do not touch mouse or
  Tab, press PgDn immediately — it should scroll.

   (`davyagent/agents/session.py`) — skill injection

7. **Role configs** (`davyagent/agents/roles.py`,

8. **Event templates** (`davyagent/events/*.md`) —

9. **Skill files** (`davyagent/skills/*.md`) —

  a) Is the system prompt clearly scoped and concise?

## 15. Smart tool execution confirmation

When the LLM requests a tool call, show a menu that
lets the user approve once, approve for the remainder
of the session (auto-approve), or deny.

### Current state
Tool calls are either auto-approved (execution mode,
`require_confirmation=False`) or gated by the plain
Yes/No prompt from item #14. There is no per-tool
persistent approval.

### Required work

#### TUI (`davy_cli.py`)
- On `tool_confirm_request` event, build the menu
  prompt:
  ```
  Run tool [<name>] with arguments <args>?
  ```
  Options:
  - `"Yes, once"`
  - `"Yes, always for this session"`
  - `"No"`
- `on_select` callback:
  - `"Yes, once"` → enqueue `True` once.
  - `"Yes, always for this session"` → record the
    `(name, args_json)` pair in a module-level dict
    `_auto_approved: dict[str, set[str]]`
    (key = tool name, value = set of
    `json.dumps(args, sort_keys=True)` strings),
    then enqueue `True`.
  - `"No"` → enqueue `False`.
- Before raising the confirmation menu, check
  `_auto_approved` for the incoming `(name, args)`.
  If a matching entry exists, skip the menu and
  enqueue `True` automatically; show a dim inline
  notification `"Auto-approved: <name>"`.

#### Session (`session.py`)
- No changes needed to `_confirm_queue` logic.
  The auto-approval lives entirely in the TUI layer.

#### State structure
```python
# module-level in davy_cli.py
_auto_approved: dict[str, set[str]] = {}
# key: tool name
# value: set of JSON-serialised arg dicts
```
Cleared on session reset (`/clear` or new session).

### Notes
- `args` must be serialised with `sort_keys=True`
  before storing so key order does not affect
  matching.
- Auto-approval is session-scoped only (in-memory).
  It is NOT persisted to the session JSON on exit.
- The menu title should include a short preview of
  the args (truncated to 60 chars) so the user can
  make an informed choice.

## 16. Unknown slash command — error + help

When the user types a `/` command that is not
recognised, show an error frame and the help text
instead of silently sending the string to the LLM.

### Current state
Unrecognised `/foo` input falls through to the
default LLM dispatch path at the bottom of
`accept_handler`. The LLM receives `/foo` as a
user message with no explanation.

### Required work
- In `accept_handler`, after all recognised command
  branches and before the default LLM dispatch,
  add a guard:
  ```python
  if text.startswith("/"):
      # unknown slash command
  ```
- On an unknown `/` command:
  1. Append the user frame with the typed text
     (so the command is visible in the buffer).
  2. Show an error frame:
     `"Unknown command: '<text>'. "
      "Type /help for a list of commands."`
  3. Show the same help AI frame already rendered
     by `/help` (extract the help_md string into a
     module-level constant or helper so it is not
     duplicated).
  4. Do NOT dispatch to the LLM.
- Extract `HELP_MD` as a module-level constant
  (or a `_build_help_md()` function) shared by
  both the `/help` handler and the unknown-command
  fallback.

### Recognised commands (as of this plan)
`/exit`, `/clear`, `/help`, `/tools`, `/skills`,
`/sessions`, `/notify`, `/ask`, `/mode`, `/role`,
`/setup`, `/init`

### Notes
- The check must be `text.startswith("/")` (case-
  sensitive prefix), not `text.lower()`, so that
  `/Help` or `/SETUP` also triggers the error.
  Recognised commands already use `text.lower()`
  in their own `if` guards, so they match before
  reaching this fallback.
- Prefix commands like `/notify <message>` are
  already matched with `startswith`, so they are
  not affected.
