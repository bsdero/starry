## 1. question tool — TUI input collection

When the LLM calls the `question` tool, the TUI must pause
and present an interactive input prompt to the user.

### Current state
`question.execute()` returns a `user_input_required` stub
dict. The TUI renders a warning frame but does not collect
real user input.

### Required work
- Detect `tool_result` events whose `name == "question"` and
  whose result contains `user_input_required`.
- Pause the tool loop (do not re-submit to the LLM yet).
- Show a dedicated inline input prompt in the TUI.
- On submission, inject the typed text back into the tool
  result and resume the LLM loop.

- Surface the child session's events (or at minimum the

## 8. Event system (davyagent/events/)

Custom markdown responses sent to the LLM when specific
CLI events occur. Allows the LLM to stay informed about
user actions that happen outside the normal chat flow.

### Event files

Each event is a `.md` file in `davyagent/events/`.
Files support `{{variable}}` template placeholders that
are substituted at fire time.

| File                          | Fires when                          | Variables available              |
|-------------------------------|-------------------------------------|----------------------------------|
| `on_llm_request_cancel.md`    | User presses Escape on an LLM-      | —                                |
|                               | driven options menu                 |                                  |
| `on_session_start.md`         | A new session is established        | `{{session_id}}`, `{{role}}`,    |
|                               |                                     | `{{provider}}`, `{{model}}`      |
| `on_session_end.md`           | User cleanly quits the TUI          | `{{session_id}}`,                |
|                               |                                     | `{{message_count}}`              |
| `on_tool_call.md`             | Just before a tool executes         | `{{tool_name}}`, `{{tool_args}}` |
| `on_tool_error.md`            | A tool execution raises an          | `{{tool_name}}`,                 |
|                               | exception                           | `{{error_message}}`              |
| `on_mode_change.md`           | User switches plan/execution mode   | `{{old_mode}}`, `{{new_mode}}`   |
| `on_error.md`                 | Generic LLM/API error               | `{{error_message}}`              |
| `on_context_limit.md`         | Conversation history approaches     | `{{token_count}}`,               |
|                               | the model's token limit             | `{{token_limit}}`                |

### EventLoader — `davyagent/events/loader.py`
- `load_event(name, **kwargs) -> str | None` — reads the
  corresponding `.md` file, substitutes `{{var}}` placeholders
  with `kwargs`, returns the rendered string. Returns `None`
  if the file does not exist (event silently skipped).
- Missing variables in the template are left as-is (no crash).

### Session integration
- `Session.fire_event(name, **kwargs)` — calls `load_event`,
  and if a string is returned, injects it as a `role="user"`
  message into the active conversation (not shown in TUI
  chat history, marked internal).

### TUI integration
- Escape key on any LLM-driven menu: call
  `session.fire_event("on_llm_request_cancel")` then
  re-submit the conversation so the LLM acknowledges the
  cancellation.
- Mode toggle (Ctrl-P or `/mode`): fire `on_mode_change`.
- All existing error paths: fire `on_error` or
  `on_tool_error` as appropriate.

## 9. Skills (davyagent/skills/)

Skills are markdown files that provide the LLM with
specialised context or instructions for a task domain.
The user selects a skill from an options menu; it is
injected as a system message and remains active for the
session.

### /skills command
- Shows a scrollable options menu listing all `.md` files
  found in `davyagent/skills/` (filename without extension
  = skill name).
- Escape key on the menu: fires `on_llm_request_cancel`
  event; menu closes, nothing changes.
- On selection: reads the file, appends its content as an
  additional system message block to the active session.
- Multiple skills may be active simultaneously; each is a
  separate system message block.

### SkillLoader — `davyagent/skills/loader.py`
- `list_skills() -> list[str]` — returns names of all `.md`
  files in `davyagent/skills/`.
- `load_skill(name) -> str` — reads and returns the file
  content. Raises `FileNotFoundError` if absent.

### Session integration
- `Session.active_skills: list[str]` — tracks currently
  loaded skill names.
- `Session.add_skill(name)` — loads skill via `SkillLoader`,
  appends to `active_skills`, injects into system messages
  for subsequent LLM calls.
- `Session.remove_skill(name)` — removes from active list.
- Skills are included in persisted session state (item 10).

## 10. Session persistence (davyagent/sessions/)

Sessions are serialised to disk on clean exit and can be
fully restored via the `--session` CLI flag.

### Session file
- Path: `davyagent/sessions/session_<hash>.json`
- Hash: SHA-256 of `session_id` string, first 12 hex chars.
- Format: JSON. Contents:

```json
{
  "session_id": "...",
  "hash": "...",
  "created_at": "ISO-8601",
  "saved_at": "ISO-8601",
  "provider": "...",
  "model": "...",
  "role": "...",
  "mode": "plan|execution",
  "active_skills": ["skill_a", "skill_b"],
  "history": [
    {"role": "user|assistant|system", "content": "..."}
  ]
}
```

### Save / restore
- Save: triggered on clean TUI exit (Ctrl-C handler,
  `/exit` command). Writes the file atomically
  (write to `.tmp`, then rename).
- `SessionStore` — `davyagent/sessions/store.py`:
  - `save(session: Session) -> Path`
  - `load(hash_prefix: str) -> dict` — accepts full hash
    or unambiguous prefix.
  - `list_sessions() -> list[dict]` — returns metadata for
    all saved sessions (id, hash, saved_at, role, provider).

### CLI restore
- `davy_cli.py --session <hash>` — loads the session file,
  restores history, provider, model, role, mode, and active
  skills, then starts the TUI in that session context.

### /sessions TUI command
- Lists all saved sessions (hash, role, provider,
  saved_at, message count).
- User can select one to resume it in the current process
  (history is appended to current session or a new session
  is spawned from the saved state).
- Escape key closes the menu without action; fires
  `on_llm_request_cancel`.

### Quit behaviour
- When the user exits (cleanly), the TUI displays:
  ```
  Session saved.
  Resume with: davy_cli.py --session <hash>
  ```

All TUI colours are driven by a JSON theme file. No colour
value is hardcoded in `davy_cli.py`.

### Theme file format — `davyagent/themes/default.json`

Keys map to the current hardcoded constants. Example:

```json

  "sys_text":          "#ff9e64",
  "accent_lime":       "#9ece6a",
  "accent_fuchsia":    "#bb9af7",

  "user_frame_bg":     "#1a1f2e",

  "warning_frame_bg":  "#1e1a10",
  "top_bar_bg":        "#1e253c",

  "md_bold":           "#e0af68",
  "md_code":           "#9ece6a",
  "md_code_bg":        "#1e253c"

```

### ThemeLoader — `davyagent/themes/loader.py`
- `load_theme(name: str = "default") -> dict` — reads
  `davyagent/themes/<name>.json`, returns the colour map.
- `list_themes() -> list[str]` — returns names of all `.json`
  files in `davyagent/themes/`.
- Active theme stored in a module-level dict; `davy_cli.py`
  reads colour values from it via `theme["key"]`.

### davy_cli.py changes
- Replace all hardcoded hex strings with `theme[key]`
  lookups at startup.
- Build the `prompt_toolkit` `Style` object from the loaded
  theme dict after `load_theme()` is called.

### /setup — theme submenu
- Add a "Theme" option to the existing `/setup` menu.
- Opens a submenu listing all available themes from
  `list_themes()`.
- On selection: calls `load_theme(name)`, rebuilds the
  `Style` object, and calls `app.invalidate()` to redraw.
- Escape key on the submenu: fires `on_llm_request_cancel`,
  closes submenu, active theme unchanged.

### Escape key support for all built-in menus
- All option menus in the TUI (setup, /skills, /sessions,
  theme submenu, tool confirmation) must handle the Escape
  key uniformly.
- On Escape: close the menu, fire
  `session.fire_event("on_llm_request_cancel")` if an LLM
  interaction was pending, otherwise just dismiss silently.

## 12. Mode-dependent user prompt colors

User input text and user message frames in the scrollable
area change color to reflect the active execution mode.

### Theme file additions

Two new color identifiers added to every theme file
(e.g. `default.json`):

```json
"text_mode_plan":      "#c0caf5",
"text_mode_execution": "#e59d56"
```

- `text_mode_execution` — `#e59d56`
- `text_mode_plan` — `#c0caf5`

### Accent color rename

Rename in all theme files and throughout the codebase:

| Old key         | New key    |
|-----------------|------------|
| `accent_lime`   | `accent_1` |
| `accent_fuchsia`| `accent_2` |

All references in `davy_cli.py` (style dict and any
direct usage) must be updated to the new names.

### Python constants (davy_cli.py)

Following the existing short-caps convention
(`SYS_TEXT`, `BG_DEEP`, etc.), derive two constants
from the loaded theme at startup:

```python
MODE_CLR_EXEC = theme["text_mode_execution"]
MODE_CLR_PLAN = theme["text_mode_plan"]
```

### Affected style classes

`line.uframe` and `line.ucontent` currently use
`SYS_TEXT` (#ff9e64) for both frame border and
content text. These must become dynamic:

| Style class     | Execution mode    | Plan mode         |
|-----------------|-------------------|-------------------|
| `line.uframe`   | `MODE_CLR_EXEC`   | `MODE_CLR_PLAN`   |
| `line.ucontent` | `bold MODE_CLR_EXEC` | `bold MODE_CLR_PLAN` |

The live input area (`input-area` style) also switches:

| Style key    | Execution mode                        | Plan mode                             |
|--------------|---------------------------------------|---------------------------------------|
| `input-area` | `{MODE_CLR_EXEC} bg:{theme[bg_deep]}` | `{MODE_CLR_PLAN} bg:{theme[bg_deep]}` |

### Mode switch wiring

Whenever the mode changes (Ctrl-P, `/mode` command, or
`/setup` mode option), the Style object must be rebuilt
with the new mode colors and `app.invalidate()` called
to redraw immediately. A helper:

```python
def build_style(mode: str) -> Style: ...
```

replaces the current static `Style.from_dict(...)` call
and is invoked at startup and on every mode switch.
